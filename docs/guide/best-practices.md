# 建議用法

OffiCraft 給你的東西很簡單，但要用得好、讓成員愈幫愈上手，有幾個習慣值得養成。這一頁講**怎麼把它用出價值**：
把手冊寫好、在對的時機開卡、該外包時外包、把風險與成本的界線畫清楚。

想先理解任務的設計原理，看 [任務是怎麼運作的](tasks.md)；名詞不熟，開 [名詞表](glossary.md)。

---

## 一、把重複的事沉澱成任務手冊

**一次性的事直接交給成員做掉；反覆出現的同一類事，值得建一本任務手冊。**

手冊是這一「類」工作的標準流程——用途、要哪些欄位、該怎麼做（SOP）。建一次、跨任務長期存在，而且**愈用愈準**：
每張結案任務都會把教訓寫回同一本手冊，下一個接手的人開工前先讀。這是 OffiCraft 複利的地方——**同樣的交代不必講第二次。**

判準很簡單：**同一型請求出現第三次、你發現自己在重講一樣的交代**，就該建手冊了，別讓經驗散在一張張零散任務裡。

### 手冊裡最關鍵的一格：識別鍵

手冊會問你「哪個欄位是這件事的身分」——這就是**識別鍵**，防重複的根基。同一個識別鍵值（例如同一個 PR 連結）
算同一張任務，系統據此自動去重、不重複開單。

> **挑識別鍵的判準：選「這件事是什麼」，不要選「這次是怎麼進來的」。**
> review 一個 PR，身分是那個 PR，識別鍵就只有 `pr_url`。如果把「從哪串 Slack thread 進來的」也放進識別鍵，
> 同一個 PR 從另一串對話被提起、或你直接在控制台貼連結（根本沒有 thread），就會被當成不同的事而開出第二張卡——
> **防重複剛好在最該生效的時候失效。**

（識別鍵可以複合，但複合的每一欄都必須是身分的一部分，例如「同一個 repo 的同一個版本號」。識別鍵欄位一律必填。）

> **提醒：手冊「由誰負責」只有你（owner）能設。** 那是治理面的設定——某類任務歸哪個成員、要不要外包、跑幾份、
> 在哪台機器——成員自己改不了。要綁負責成員或改成外包，去 **設定 › 任務手冊** 設。

## 二、寫好 DoD：給 AI 一把你自己的槓桿

每個節點都要有**完成定義（DoD）**——「怎樣算做完」的那句話。這是整套設計最省你力氣的一環，值得寫好。

**好的 DoD 是「可觀察的結果」，不是「要做的動作」：**

| ❌ 不好 | ✅ 好 |
| --- | --- |
| 跑一下測試 | 全部測試通過，且新增的三條護欄各自實測會擋 |
| 處理好 PR | PR 上留下 approve 或 request-changes 的結論 |
| 等部署完成 | 已發佈到 registry、健康檢查通過 |

**判準只有一個：讀完這句話，任何人都能客觀判斷它有沒有達成。** 做不到就重寫。一個 DoD 只講一個結果——
塞兩件事進去就會出現「一半做完」這種無法判定的狀態，那就該拆成兩節。

為什麼值得講究？因為**沒有完成定義的工作，AI 一定會回報「做好了」**，而你得自己去查它到底做了什麼。
DoD 把「做好了」變成一句可以驗證的話——**它不是給 AI 的規矩，是給你的槓桿。**

## 三、什麼時候該開卡（gate），什麼時候讓它跑

成員需要你決定時會開一張**等我回覆卡**停下來等。你這邊要拿捏的是：**哪些事值得停下來問你、哪些該放手讓它跑完。**

成員自己遵循的判準是「**卡在你身上、而且非你不可**才開卡」——查得到的事實它自己查、能自己做的動作它自己做，
只有「只有你能決定／批准／提供」的事才來問你。你可以順著這個心智模型去設計工作：

**該讓它成為 gate（停下來問你）的：**

- **有代價、不可逆、或會踩到風險底線的決定**：要不要寄出、要不要 merge、能不能動線上、要不要花這筆。
- **方向選擇**：AI 驗收不了、兩個方向都合理、得你拍板選一個。
- **超出原本交代範圍的延伸**：交代的做完了，它想順手多做你沒要求的事——這需要你 go/no-go。

規劃任務時，把這些節點**預先標成 gate**，好處是控制台會**在計畫送出當下就預告**「這件事後面要你點頭一次」——
你一眼就知道整件事會在哪裡需要你，不會被半途殺出的問題打斷。

**該讓它自己跑完、別開卡的：**

- 純粹回報進度、確認收到、FYI——這些走一般聊天就好，不該佔用你的 Ask 收件匣。
- 它查得到、算得到、試得出來的事——讓它自己解，別讓它把功課丟回給你。

> **拿捏原則：寧可讓 gate 落在「有代價或不可逆」的關口，別散落在雞毛蒜皮上。** gate 開太密，你就變成人肉
> 迴圈、失去「不需要盯著螢幕」的好處；gate 開太疏，它可能在你其實想管的地方自己決定了。
> 把 gate 對準**真正只有你能拍板、而且錯了要付代價**的那幾步。

## 四、什麼時候該外包

**外包（outsource worker）是臨時開、做完就收的匿名工作者，一個綁一張任務。** 跟常駐成員最大的差別：外包
**可以自己下場做重活**（開發、測試、研究都自己來），做完就退場解僱——它的 context 用完即棄，不必守著長期預算。

適合外包的：

- **可以獨立切出去、做完即結案的一整包工作**——例如「review 這一個 PR」「把這份資料整理成報告」。
- **你想並行推很多份同類工作**：一次發包多張，滿額就排隊、有名額自動接，不必你逐張催、也不必逐筆核可
  （同類的並行份數受該類手冊的**雇用份數**上限約束，**全域同時雇用上限**則是最終兜底）。
- **不需要跨任務累積角色記憶的一次性活**——外包沒有角色、沒有角色學習筆記，它的經驗回寫進**任務手冊**。

留給常駐成員的：**需要長期記憶、需要協調多件事、需要一個穩定角色持續深耕的工作**（例如你的助理 Mira，
或某個專責某個 repo 的角色）。常駐成員的主 session 是協調者，重活它會自己開 sub-agent 或發外包去做。

> 成本由**全域同時雇用上限**兜底——外包開太多不會無限燒，滿額就排隊。這個上限 owner 可以在控制台調。

---

## 五、什麼時候該加一個常駐成員

外包是「做完就收」的另一面，這一面是：**什麼時候值得請一個常駐成員、養一個角色。** 常駐成員跟外包最大的差別是
**它有身分、有跨任務累積的角色記憶與學習筆記**——同一個角色一手接一手長期存在（見 [架構與運作原理](architecture.md)
的換手一節）。養一個角色是有成本的（它要占一個常駐名額、要你把它的守則交代清楚），所以值得在對的時機開，別為了
一次性的活開常駐。

適合開一個常駐成員的：

- **一整條領域、或一個 repo，需要長期記憶與角色深耕**——它會愈做愈熟這塊地：踩過的坑、你的偏好、這個 repo 的
  眉角，都沉澱進它的角色記憶，下一手不必重學。
- **某類協調反覆出現、需要一個穩定的窗口**——像你的助理 **Mira**（幫你分派、追進度、擋雜事），或一個專責發版的
  角色。這類工作的價值在「總是同一個人接、接口穩定」，開常駐才留得住那份熟悉。
- **希望跨世代累積角色學習筆記的工作**——你要的是一條愈走愈準的學習曲線，而不是每次重來。角色記憶跨 session
  存在 server 上，正是為這種複利設計的。

這些情況**不必**開常駐、外包就好：

- **一次性、做完即結案的活**——沒有下一手要接，累積角色記憶沒有意義。
- **你想並行推很多份的同類工作**——那是外包的主場（受該類雇用份數與全域同時雇用上限兜底），不需要一個穩定角色。
- **不需要跨任務累積角色記憶的工作**——它的經驗回寫進**任務手冊**就夠了，不需要一個角色來承載。

> **判準一句話：需要「同一個角色長期接、記憶愈疊愈厚」就開常駐；只是要把一包事做掉、做完就散，就發外包。**
> 常駐成員的主 session 是協調者，真正的重活它會自己開 sub-agent 或發外包去做——所以「開常駐」不等於「這個角色
> 什麼都自己扛」。

## 六、把風險與成本的界線畫清楚（風險閘）

這是最重要、也最需要你親自定的一件事。成員的判斷有一條最上層的優先順序，叫**風險閘**：

> **風險底線只有你能定。** 還在你能接受的範圍內就放手做；一旦會踩線——超支、造成無法挽回的損害
> （弄壞線上、刪資料、洩漏機密）、選錯難回頭的大方向——就停下、先問你。**這一條過不了，其他都免談。**

成員被設計成**誠實暴露風險**（不報喜不報憂、拿不準就說拿不準）、並且**守在你畫好的邊界內**。但邊界那圈籬笆是
**你**畫的——旋鈕在你手上。要讓風險閘真的有用，你這邊要做的是：

- **把邊界講清楚**：哪些 repo 能碰、哪些絕對不能動、什麼要先問、預算大概多少。這些可以寫進 **設定 › 角色誌** 的
  **「使用者自訂」全域情境**（那一塊你可編輯），成員每次開機都會讀到。
- **把「要付代價的決定」設成 gate**（見「三、什麼時候該開卡」）——讓風險閘有一個具體的停下來的機制。
- **相信「停下來問」這個機制，而不是系統權限牆**。要記得一件事——

> 🔴 **成員的 Claude Code 是用 `--dangerously-skip-permissions` 跑的。** 這是刻意的（無人看管的背景 session，
> 跳權限框沒人會按）。代價要講明白：**成員在你的機器上有很大的行動自由**，護欄靠的是它讀到的工作守則、
> 以及「該問你的事會停下來開卡」這個機制，不是一道系統權限牆。**這件事你要接受了再用。**

### 成本剎車

成員也會顧成本：token 用量逼近額度預算線時，它會**自己放慢、變省**（少開平行 sub-agent、挑最重要的做），
不用你去踩剎車。你在控制台 **監控** 頁一眼看得到誰在哪台機器、用哪個模型、花多少、5 小時與 7 天的額度還剩多少；
7 天視窗過熱會標紅點。**成員用的是你自己的 Claude 額度——帳單也是你的**，所以這頁值得偶爾掃一眼。

## 七、讓成員一直很有用

- **一開始就把「你是誰、專案在做什麼、哪些事不能碰」寫進使用者自訂全域情境**，之後不必每次重講——這正是 OffiCraft
  相對「開一堆終端機各跑一個 Claude Code」最省事的地方。
- **讓它把教訓沉澱回手冊與角色記憶**：結案時的學習經驗回寫是自動的，你要做的是別去打斷這個累積（例如別為了乾淨
  一直重建手冊）。**換人不換記憶**——角色記憶跨 session 存在 server 上。
- **善用換手，不要怕它**：成員記憶快滿時會自動換手、無縫接上；你也可以在成員面板按 **Refocus** 主動讓它壓縮記憶
  重開一手。換手門檻可在 **設定 › 參數調整** 調（自動換手門檻 40–90%）。
- **交出去的是「一件事」，不是「一步一步的指令」**：把整件事連同「怎樣算做完」一起交出去，讓它自己拆、自己推、
  卡住了停下來問你——這比你回合制地下每一步指令省力得多，也是這套工具的設計初衷。
