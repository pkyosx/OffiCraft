# AI 工作室 — 產品 Spec

> 本文件依 UI 原型（同目錄 `docs/design/M1/AI Office M1 (standalone).html`）逐區定義。
> 可編輯的 .md 文件（全域情境、角色定義）一律共用同一套編輯模式：**編輯 → 取消編輯／重置／完成編輯**（重置＝還原預設）。
> 原則：**只寫 UI 實際出現的功能**；程式裡有 code 但畫面上點不到的，一律不納入。
> 這是一份會持續補充的 spec，不以版本切分。

## 導航總覽

主導航兩個主頁，外加右上角設定：

- **辦公室（Office）** — 與成員聊天、管理成員
- **監控（Monitor）** — 帳號用量、機器資源、AI 會話
- **設定** — 軟體更新、角色誌（含全域情境）
- **個人檔案（Profile）** — 右上角下拉

開箱時整個工作室只有：**你（CEO）** ＋ **Mira（助理）** 一位成員。

## 核心名詞

### 成員（Member）
工作室裡的 AI 成員。欄位：

| 欄位 | 說明 |
|---|---|
| 名字 | 顯示名稱（例：Mira）|
| Member-ID | 系統產生的成員編號，格式 `MB-XXX###` |
| 角色 | 擔任的角色（例：助理）|
| 狀態 | 離線 / 喚醒中 / 線上 |
| 上次上線 | 離線時顯示「N 分鐘前」|
| 模型 | 使用的 AI 模型 |
| 思考強度（Effort）| 低 / 中 / 高 |

### 成員狀態機
presence 是 **server 端的投影**，不是 agent 自報的心跳狀態。核心錨點只有兩個變數：
1. **online = SSE 連線投影**（`online == connected`）——server 依 agent 是否「正持著 `ocagent listen` 的 SSE 連線」直接投影：持著＝online、SSE 一斷＝offline。**沒有 agent 要自己維持的 heartbeat**（舊設計的 heartbeat 已移除）。**warden 亦同一套**——warden 自己連上 SSE ＝ online（2026-07-10 起 member 與 warden 一視同仁；權威 spec 見 `docs/design/state-model.md`）。
2. **喚醒中（waking）** 由 boot 起手一次性的 `ocagent presence --phase waking`（獨立 HTTP、不掛 SSE）錨定，發生在掛 listen **之前**。

其餘態由 server 從這兩個錨點 ＋ graceful-shutdown 訊號**衍生**（單一衍生函數 `presence_state()`）：
```
離線(offline) ──(你觸發喚醒；UI 不改狀態，仍離線)──►
   ──(agent 報 waking：presence --phase waking)──► 喚醒中(waking)
   ──(agent 掛上 ocagent listen、server 見 SSE 連上)──► 線上(online == connected)
   ──(收 stopping 訊號、SSE 仍在)──► stopping ──(SSE 斷)──► stopped/offline
```
- **離線 / stopped**：灰點
- **喚醒中（waking）**：黃點 ＋ `喚醒中` 徽章（已報 waking、SSE 尚未連上）
- **線上（online）**：綠點（server 見 agent 正持著 SSE 連線——純連線投影，非 heartbeat）
- **stopping**：winding-down 內部衍生態（graceful-shutdown 訊號已設、SSE 未斷），由 server 算出，agent 不用管。

> **單一 presence 信號、單一畫點處。** 成員的狀態點由 **PresenceBadge / LifecycleDot** 這一個 presence 信號源畫；**Avatar 本身不畫點**（頭像只是頭像），避免兩處各畫一顆點而彼此不同步。

### 全域情境（Global Context）
**套用到所有成員**的 boot context，拆成三塊（依組裝順序）：**系統互動**（系統運作說明，唯讀 seed）→ **使用者自訂**（owner 可編輯的 additive 追加塊，空時整塊跳過）→ **啟動程序**（工作室固定 SOP，唯讀 seed）。於「設定 › 角色誌」檢視；只有「使用者自訂」可編輯（唯讀兩塊 backend 無寫入 endpoint，by construction）。UI 一律**不顯示 .md 檔名**（三塊是內容、不是檔案）。

## 1. 辦公室（Office）

預設首頁。版面：左＝成員列、右＝對話區（窄螢幕切單欄）。

### 1.1 頂部列
- **左**：公司標誌 ＋ 組織名稱 ＋ 鉛筆編輯（Enter／失焦儲存）。標誌只是品牌顯示，此區**無點擊行為**（不會開啟聊天）。
- **右**：重新整理、設定齒輪、個人檔案下拉。

### 1.2 成員列
- 標題：`辦公室成員 · {人數}`
- 每張成員卡片顯示：頭像＋狀態點、名字、`離線`徽章（離線時）、角色名、上次上線（離線時）、**「聊聊」按鈕**
- 點卡片本身 → 進入該成員的**詳情面板**；點「聊聊」→ 開該成員聊天室

### 1.3 對話區（聊天）
- 與選定成員一般對話：輸入訊息、送出
- 可**附加檔案／圖片**給 AI 參考（支援直接貼上圖片）
- 空狀態文案：「這個範圍還沒有訊息」

### 1.4 成員詳情面板
從成員卡片點入，頂部有「返回」。

**身分卡**
- 頭像、名字、**Member-ID** 徽章、**改名**（鉛筆 → 內嵌輸入）
- 狀態列：狀態點 ＋「角色 · 狀態」
- 動作：**喚醒**（離線時）／**手動喚醒**（線上時）

**資訊卡**
1. **模型** ｜ **Effort 思考強度**（低/中/高）
2. **運行狀況**：機器·帳號（待命時顯示「待命中」）；🧠 **context %**（旁有 **重新聚焦** 按鈕）；💲 **估計$**
3. **終端 · TMUX**：`$ tmux attach -t <id>` 指令 ＋ 複製鈕（在自己的終端貼上即可接上該成員 session）
4. **初始 prompt**（可展開）：喚醒該成員時注入的角色 prompt
5. **學習經驗**（可展開）

### 1.5 喚醒流程
面板按「喚醒」→ **直接在本機啟動 agent**（一鍵，無選擇視窗）。

喚醒動作本身**不會改變成員狀態**；狀態是 server 端投影（見「成員狀態機」）：
- agent boot 起手報 `presence --phase waking` → 轉 **喚醒中**
- agent 掛上 `ocagent listen`、server 見 SSE 連上 → 轉 **線上**（`online == connected`，純 SSE 連線投影，無 heartbeat）

狀態為「喚醒中」時，喚醒按鈕**照常顯示、不做 disable**。

### 1.6 重新聚焦（Refocus）
壓縮／整理成員的 context（context 快滿時瘦身）。只有**線上**時可按，位於 context% 旁。

**偏好子頁**

## 3. 設定

齒輪進入。landing 標題「設定」，兩個入口：

### 3.1 軟體更新
麵包屑 `設定 › 軟體更新`。一張卡片：
- **目前版本**（例：v1.2.0）
- 若已最新：「✓ 已經是最新版」
- 若有新版：「有可用的新版本 v1.3.0」＋ **升級按鈕**
- 只顯示版本資訊與升級，不重複顯示組織名稱

### 3.2 角色誌
麵包屑 `設定 › 角色誌`。分兩區：

**區塊 1 — 全域情境（GLOBAL CONTEXT）：三塊**
- 角色誌列表依 boot 組裝順序列三列（**不顯示 .md 檔名**）：
  - **系統互動**（`系統唯讀` 徽章）：點進 → 唯讀詳情頁，只有渲染後的 markdown，**無編輯入口**
  - **使用者自訂**：點進 → 詳情頁：檢視 markdown；**編輯** → 編輯模式（**取消編輯／重置／完成編輯**）；**重置**＝清空（回到未撰寫的 `預設` 空塊；boot context 組裝時空塊整塊跳過）
  - **啟動程序**（`工作室 SOP` 徽章）：點進 → 唯讀詳情頁，**無編輯入口**

**區塊 2 — 角色定義**
- **列表**：角色卡片，每張顯示 頭像、**角色標題**、描述（目前只有「助理」一張）。點卡片 → 角色詳情。
- **詳情頁**：
  - **角色標題**（頁面大標題）
  - **職責說明**：該角色的描述
  - **角色定義 .md**：檔名 ＋ 檢視 markdown；**編輯** → 編輯模式（**取消編輯／重置／完成編輯**，重置＝還原預設角色定義）— 該角色的行為定義

## 4. 監控（Monitor）

一頁三區塊（即時遙測資訊；欄位如下，資料接真實遙測）。

### 4.1 帳號資訊（Accounts）
每帳號一張卡：
- 狀態點（綠／紅=過熱）、**帳號名**、`· 機器名`、右側 **估計 $花費**
- **5 小時窗**：`· 用量 X% · 時間 Y%` ＋ 進度條（填色＝用量、標記線＝時間進度）
- **7 天窗**：同上；用量燒太快（超過時間 >5%）標 **`· 過熱`**

> 比較「用量 %」與「時間流逝 %」，燒太快就示警（Claude 訂閱用量窗口）。

### 4.2 機器資訊（Machines）
表格，一列一台機器：
**機器** ｜ **帳號** ｜ 🖥 CPU ｜ 🎚 RAM ｜ 🔋 電量 ｜ 🔌 電源

### 4.3 AI 會話（AI Sessions）
表格，一列一個運行中的成員 session（**點列 → 開該成員詳情面板**）：
**成員**（頭像＋名字＋狀態點＋`角色 · 狀態`）｜ **機器** ｜ **帳號** ｜ **模型**（＋effort 徽章）｜ 🧠 context ｜ 💲 估計$

> **effort 徽章＝真實即時 reasoning-effort（非 owner-intent）。** Sessions 這欄的 effort 是 agent **當下**的 reasoning-effort，走既有 token-usage telemetry 便車、跟 model / cost 同源同路：
> **statusLine（`effort.level`，反映 session 中途 `/effort` 變更）→ `_build_telemetry`（agent 端組包）→ `POST /api/monitoring/telemetry` → `MonitoringSessionDTO.effort` → FE `session.effort`**。honest-empty：尚未上報時是 `""`，UI 顯示破折號「—」，不假造。
> 對照：成員詳情面板（§1.4 資訊卡）的 effort 是 **`member.effort`（owner-intent，寫死 `medium`）**——那是「想要多少」的意圖值、不是即時值。兩者刻意分流：Sessions 看真實即時、詳情面板看意圖。**spawn 仍 pin `--effort medium`**（launch intent），但那只是起手預設，≠ 即時顯示——中途 `/effort` 一改，telemetry 就上報新值、Sessions 徽章即時跟著變。
